## 3. 人工观察结论

- 样本 `微信图片_20260616093037_3804_37.jpg`：
  - 整页 OCR 能覆盖大量印刷题干、选项、页眉和部分手写答案，说明 PaddleOCR 在清晰印刷英文和较明显手写区域上有基础可用性。
  - 反面透字、纸张折痕、阴影、页面边缘和密集排版产生了不少干扰框。
  - 手写答案只适合做候选识别结果，不能直接作为判分依据。
  - 同一页包含多栏、多区域和多种题型，整页 OCR 输出缺少题区归属关系。
- 样本 `微信图片_20260616093039_3805_37.jpg`：
  - 整页 OCR 同样能识别大量印刷文字和部分手写内容。
  - 图片整体横置，OCR 仍能检测文本框，但阅读方向和题区组织混乱。
  - 页面旋转、题区定位、答案区裁剪必须先做，否则 JSON 中的文本顺序不能直接用于判分。
  - 选择题选项、阅读材料、学生手写答案和背面透字会混在同一批 OCR 结果里。

结论：整页 OCR 适合作为“看问题在哪里”的调试手段，不适合作为最终判分输入。下一步必须走“页面方向矫正 -> 模板对齐 -> 题区裁剪 -> 小区域 OCR”的路线。

## 4. 运行中发现的工程问题

- PaddleOCR 3.7.0 默认会加载文档方向、去畸变、文本行方向等额外模型。当前最小验证先关闭这些非必要模型，避免下载时间过长，也避免无关模型影响第一轮验证。
- 当前项目路径 `F:\智能批阅机` 含中文，Paddle 底层读取模型文件时可能在中文缓存路径下报 JSON 解析错误。已调整 `scripts/setup-env.ps1`：当项目路径包含非 ASCII 字符时，模型缓存自动切到 `F:\aipiyue_cache`，学生试卷图片和输出文件仍留在项目目录。
- PaddleOCR 在 CPU 上默认启用 MKLDNN/oneDNN，本机组合下会报 `ConvertPirAttribute2RuntimeAttribute not support`。已在 `PaddleOcrEngine` 中默认关闭 `enable_mkldnn`，保证 CPU 验证链路能跑通。
- 两张样本原图分辨率较高，OCR 会把长边压到 4000 内部处理，但整页识别仍然较慢。后续应在预处理中增加可配置的最大边长，先服务调试效率。

## 5. 下一步工程方向

下一步不直接用整页 OCR 判分，而是进入模板对齐、题区裁剪、小区域 OCR、规则判分和人工复核流程。

建议下一批任务：

1. 增加页面方向检测和手动旋转参数，优先处理横置样本。
2. 增加预处理输出最大边长配置，降低 CPU OCR 调试耗时。
3. 基于答案卷或老师框选，先做一版手工模板 JSON。
4. 按模板裁剪 3 到 5 个小题区域，比较小区域 OCR 与整页 OCR 的稳定性。
5. 为选择题、判断题、填空题分别设计规则判分的最小输入格式。
