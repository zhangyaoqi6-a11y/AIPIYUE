# OCR 验收网页设计

日期：2026-06-24

## 1. 目标

下一阶段先做一个本地网页，让验收时能看到真实识别效果，而不是只看文档汇报。

网页必须能展示：

- 真实样本图片。
- OCR 识别框。
- OCR 文字、置信度和位置。
- 自动抽取的题目候选。
- 每个候选项的人工验收状态：待确认、正确、错误。

第一版只服务本地验收，不做账号、班级、学生名单和最终判分。

## 2. 方案选择

推荐方案：FastAPI 本地验收网页。

- 后端直接读取 `data/outputs/minimal-ocr/*.ocr.json` 和图片文件。
- 前端使用普通 HTML、CSS、JavaScript，不引入 React 和构建工具。
- 人工验收状态保存到 `data/reviews/ocr-review.json`，不提交 Git。

备选方案：

- 纯静态 HTML：实现快，但不能可靠保存人工验收状态。
- Electron：更接近最终桌面端，但当前阶段会过早增加复杂度。

## 3. 页面结构

页面第一版包含三栏：

1. 左侧样本列表：展示每张真实样本、文本框数量、图片尺寸。
2. 中间图片验收区：显示预处理图，叠加 OCR 框，可切换显示框图。
3. 右侧识别结果区：展示题目候选、OCR 文本、平均置信度和人工状态按钮。

题目候选不是最终题区模板，只是根据 OCR 文本中的题号模式自动抽出的调试入口。它用于让用户快速看到“哪些题被识别出来、识别文本是什么、位置大概在哪里”。

## 4. 数据流

1. `backend/review/ocr_review.py` 读取 OCR JSON。
2. 将每个 OCR 文本框转换为网页坐标数据。
3. 使用简单规则抽取题目候选：
   - `1.`、`2.`、`3.` 这类英文试题题号。
   - `①`、`②`、`③` 这类序号。
   - `一、`、`二、` 这类中文分组标题。
4. `backend/web/review_app.py` 提供接口：
   - `GET /`：验收网页。
   - `GET /api/samples`：样本和候选题目数据。
   - `POST /api/reviews`：保存人工验收状态。
   - `GET /files/{path}`：读取本地输出图片。
5. 前端调用接口并渲染页面。

## 5. 验收标准

- 能在浏览器打开本地网页。
- 能看到两张真实样本。
- 能切换样本。
- 能在真实图上看到 OCR 框。
- 能看到候选题目列表、识别文本、置信度。
- 能把候选项标成正确或错误。
- 刷新页面后人工状态仍保留。
- 单元测试覆盖数据解析、题目候选抽取和状态保存。

## 6. 当前限制

- 题目候选是调试候选，不等于最终题区模板。
- 正确或错误由人工验收标注，不自动判断。
- 第一版不做完整判分，也不生成批改 PDF。
