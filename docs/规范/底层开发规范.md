# AI 智能批阅机底层开发规范

日期：2026-06-24

## 1. 总原则

本项目底层开发遵循以下原则：

- 优先使用成熟开源项目，避免重复造轮子。
- 优先选择可本地部署、免费、商用友好的技术。
- 学生数据默认在学校本地处理，不依赖云端模型。
- AI 识别结果必须允许老师复核和修改。
- OCR、图像处理、模型能力都必须通过适配层接入，方便后续替换。
- 第一阶段以试卷识别和批阅准确率为核心，不优先追求复杂前端功能。

## 2. 开源许可证规则

优先使用：

- Apache-2.0。
- MIT。
- BSD。

谨慎使用：

- GPL。
- AGPL。

GPL 或 AGPL 项目默认不直接嵌入主程序。确实需要使用时，必须单独评估许可证风险。

## 3. 推荐技术栈

### 3.0 开源项目选型记录

第一阶段优先采用以下开源项目：

| 项目 | 用途 | 当前定位 | 链接 |
| --- | --- | --- | --- |
| PaddleOCR | OCR 主引擎、文档识别、版面理解基础能力 | 第一主线 | https://github.com/PaddlePaddle/PaddleOCR |
| RapidOCR | 轻量 OCR、Windows 普通电脑离线部署备选 | 备选适配 | https://github.com/RapidAI/RapidOCR |
| OpenCV | 图像矫正、裁切、去阴影、模板对齐 | 必选基础库 | https://github.com/opencv/opencv |
| PyMuPDF | PDF 拆页、读取、生成批改 PDF | 第一 PDF 工具 | https://github.com/pymupdf/PyMuPDF |
| Tesseract OCR | 通用 OCR 备选 | 暂不作为主线 | https://github.com/tesseract-ocr/tesseract |
| EasyOCR | 快速 OCR 试验 | 暂不作为主线 | https://github.com/JaidedAI/EasyOCR |
| MMOCR | OCR 研究和复杂模型参考 | 暂不作为主线 | https://github.com/open-mmlab/mmocr |
| Qwen-VL | 本地视觉大模型增强 | 后期预留 | https://github.com/QwenLM/Qwen2.5-VL |
| MiniCPM-V | 本地视觉大模型增强 | 后期预留 | https://github.com/OpenBMB/MiniCPM-V |
| GOT-OCR2.0 | OCR 大模型增强 | 后期预留 | https://huggingface.co/ucaslcl/GOT-OCR2_0 |

暂不建议直接嵌入 AGPL 许可证项目作为主程序核心依赖。类似 DocLayout-YOLO 这类项目可以参考思路，但默认不进入第一阶段主线，除非后续单独评估许可证风险。

### 3.1 桌面软件

推荐：

- Electron。
- React。
- TypeScript。

用途：

- 构建 Windows 桌面软件。
- 提供上传、复核、修改、导出 PDF 等界面。

说明：

Electron 可以把网页界面打包成 Windows 桌面软件。React 用来开发操作界面。TypeScript 用来减少界面代码中的低级错误。

### 3.2 核心识别服务

推荐：

- Python。
- FastAPI。

用途：

- 提供 OCR、图像处理、模板对齐、判分、PDF 生成等核心能力。
- 桌面界面通过本地接口调用 Python 服务。

说明：

Python 是 OCR、AI、图像处理生态最成熟的语言。FastAPI 用来把识别能力封装成本地服务，方便 Electron 调用。

### 3.3 OCR 主引擎

推荐：

- PaddleOCR。

用途：

- 识别印刷文字。
- 识别题号。
- 识别标准答案。
- 识别学生答案。
- 识别题目分值。
- 辅助文档版面分析。

使用要求：

- 不允许只把整张试卷直接扔给 PaddleOCR 后结束。
- 必须先做图像矫正、模板对齐和区域裁剪。
- 优先对小区域进行 OCR，提高稳定性。
- 识别结果必须带置信度。
- 低置信度结果必须进入人工复核。

### 3.4 轻量 OCR 备选

推荐：

- RapidOCR。

用途：

- 作为 Windows 普通办公电脑上的轻量 OCR 备选。
- 当 PaddleOCR 部署过重或速度不理想时，用于替代部分 OCR 能力。

说明：

RapidOCR 更适合轻量、离线、多平台部署。第一阶段可以先预留适配接口，不一定立即实现完整替换。

### 3.5 图像处理

推荐：

- OpenCV。

用途：

- 试卷旋转矫正。
- 透视矫正。
- 页面裁切。
- 去阴影。
- 模板对齐。
- 页面匹配。
- 题区裁剪。
- 标记位置坐标转换。

说明：

OCR 效果不好时，通常不能只怪 OCR 模型。很多问题来自扫描图歪斜、阴影、纸张变形和裁切不完整。OpenCV 是提高识别效果的基础。

### 3.6 PDF 处理

推荐：

- PyMuPDF。

用途：

- 将扫描 PDF 拆成图片。
- 读取 PDF 页面尺寸。
- 生成批改标记 PDF。
- 按模板坐标输出对号、错号、分数和总分。

备选：

- ReportLab。

说明：

第一阶段优先用 PyMuPDF，减少 PDF 读写工具数量。

### 3.7 本地数据库

推荐：

- SQLite。

用途：

- 保存批改任务。
- 保存答案卷模板。
- 保存题目区域坐标。
- 保存标准答案。
- 保存题目分值。
- 保存学生卷识别结果。
- 保存老师复核后的最终结果。

说明：

SQLite 是本地文件数据库，不需要老师或学校额外安装数据库服务，适合第一阶段桌面软件。

### 3.8 文件和数据格式

推荐：

- JSON 用于保存模板结构。
- SQLite 用于保存任务和批改结果。
- JPG、PNG、PDF 作为输入格式。
- PDF 作为最终回打输出格式。

模板数据至少应包含：

- 页面编号。
- 题号。
- 题型。
- 标准答案。
- 学生答案区域。
- 批改标记区域。
- 分值区域。
- 每题分值。
- 置信度。
- 老师确认状态。

### 3.9 本地视觉大模型预留

可作为后期增强：

- Qwen-VL。
- MiniCPM-V。
- GOT-OCR2.0。

用途：

- 辅助复杂题区理解。
- 辅助识别低质量扫描图。
- 辅助解释异常识别结果。
- 后期扩展主观题或复杂填空题。

第一阶段要求：

- 不把本地视觉大模型作为核心依赖。
- 先完成 PaddleOCR、OpenCV、模板对齐的主流程。
- 模型调用接口必须可替换。

## 4. 模块边界

系统至少拆分为以下模块：

### 4.1 导入模块

负责：

- 导入 JPG。
- 导入 PNG。
- 导入 PDF。
- 将 PDF 拆页。
- 统一转换为页面图片。

### 4.2 图像预处理模块

负责：

- 旋转矫正。
- 透视矫正。
- 去阴影。
- 裁切。
- 页面清晰度检查。

### 4.3 模板生成模块

负责：

- 从答案卷生成模板。
- 识别题号。
- 识别答案区。
- 识别分值区。
- 识别批改标记位置。
- 生成可编辑模板。

### 4.4 OCR 适配模块

负责：

- 调用 PaddleOCR。
- 后续调用 RapidOCR。
- 后续调用本地视觉大模型。
- 统一输出识别文本、位置和置信度。

要求：

- 上层业务不能直接绑定 PaddleOCR 的原始输出格式。
- 必须转换为项目内部统一格式。

### 4.5 判分模块

负责：

- 选择题判分。
- 判断题判分。
- 填空题判分。
- 空题识别。
- 多答案识别。
- 同义答案或大小写规则扩展。
- 试卷模式下计算每题得分和总分。

### 4.6 复核模块

负责：

- 展示低置信度结果。
- 展示识别异常。
- 允许老师修改标准答案。
- 允许老师修改学生答案识别结果。
- 允许老师修改分值。
- 允许老师修改批改标记位置。

### 4.7 PDF 输出模块

负责：

- 生成作业批改 PDF。
- 生成试卷批改 PDF。
- 输出对号。
- 输出错号。
- 输出每题得分。
- 输出总分。
- 确保坐标与原试卷回打位置对应。

## 5. 识别流程规范

标准流程必须是：

1. 导入文件。
2. 转换为页面图片。
3. 图像预处理。
4. 页面匹配。
5. 模板对齐。
6. 区域裁剪。
7. 小区域 OCR。
8. 题型规则判断。
9. 置信度评估。
10. 人工复核。
11. 生成批改 PDF。

禁止把整张试卷 OCR 结果直接作为最终判分依据。

## 6. 复核规则

以下情况必须提示老师复核：

- OCR 置信度低。
- 学生答案为空。
- 学生答案疑似多个选项。
- 题号识别不完整。
- 分值识别不确定。
- 页面匹配失败。
- 学生卷缺页。
- 学生卷多页。
- 页面顺序异常。
- 总分计算异常。

## 7. 两种批改模式的技术差异

### 7.1 作业批改

必须支持：

- 标准答案确认。
- 学生答案识别。
- 对错判断。
- 对号和错号 PDF 输出。

不强制支持：

- 分值识别。
- 每题得分。
- 总分。

### 7.2 试卷批改

必须支持：

- 标准答案确认。
- 学生答案识别。
- 每题分值确认。
- 每题得分计算。
- 总分计算。
- 分数和总分 PDF 输出。

## 8. 第一阶段不做的事情

第一阶段不做：

- 小程序。
- 家长端。
- 学生端。
- 云端同步。
- 长期学习分析。
- 知识点自动归因。
- 错题解析生成。
- 真实打印机直连。
- 任意版式试卷全自动识别。
- 主观题自动评分。

这些功能可以后续开发，但不能影响第一阶段识别和批阅能力验证。

## 9. 后续交接要求

为了方便新对话继续开发，后续每完成一个重要阶段，都应更新文档：

- 当前实现了什么。
- 当前没有实现什么。
- 使用了哪些模型和开源库。
- 哪些样本识别成功。
- 哪些样本识别失败。
- 失败原因是什么。
- 下一步应该做什么。

重要决策必须写入文档，不只保留在聊天记录里。
